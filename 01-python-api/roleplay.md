# Roleplay Scenarios: From Manual UI to Automation

---

## Scenario 1: The Overwhelmed Platform Engineer

**Setup the scene:**

> You're a platform engineer at a growing company. It's Monday morning and you open Slack to find 12 new messages — all developers asking you to create GitHub repositories for their new projects.
>
> You sigh. Each one takes you 3-4 minutes: open GitHub, click "New repository", type the name, configure the settings, hit create, then go back to Slack and share the link. By the time you're done, it's almost lunch and you haven't done any real work.

**The pain:**

> "I'm a platform engineer, not a button clicker. I spent 45 minutes today just creating repos."
>
> "Developers are waiting on me. I'm the bottleneck. It feels terrible."
>
> "There has to be a better way — this is literally the same 5 clicks every single time."


---

## Scenario 2: The Misconfigured Repo

**Setup the scene:**

> A week after solving the bottleneck problem, a developer messages you:
>
> *"Hey, why is my repo public? Anyone on the internet can see our internal service code!"*
>
> You check. You forgot to set it to private when you created it. It's been public for 6 days.
>
> You quietly fix it, but now you're worried — did this happen to any other repos? You start clicking through GitHub one by one, checking visibility. 20 minutes later you've found two more.

**The pain:**

> "I created the repo fast, but I made a mistake because the UI has too many options and I was rushing."
>
> "There's no default that fits every case — some repos need to be public, some private. I have to think about it every time."
>
> "I want the person asking for the repo to tell me what they need, so I can't get it wrong."


---

## Scenario 3: The Team That Doesn't Exist Yet

**Setup the scene:**

> Your company is growing fast. A new backend team of 5 engineers just joined. Their tech lead messages you:
>
> *"Can you set up GitHub access for my team? They need to be grouped together so I can manage permissions on our repos all at once."*
>
> You go to GitHub. You create the team manually. Then you add each person one by one. Then the tech lead says: *"Oh, I forgot — Ana is also joining next week, can you add her too?"*
>
> Two days later, another team forms. Same process. Then a third. You're spending a growing portion of your week just managing group membership in a UI.

**The pain:**

> "Adding people to GitHub teams takes forever when it's one click per person. Five engineers means five separate operations."
>
> "Every new team is the same manual work. I need a way to do this that scales."
